# SKILLS.md — Skills Exercised During Development

This document lists the engineering skills that were exercised — by both
the human and the AI assistant — to deliver this assignment.

## 1. Backend engineering
- **Spring Boot 3.2 application bootstrap** — entry point, configuration
  properties binding, environment-driven secrets.
- **Layered architecture** — controller / service / repository / entity
  with DTOs for the boundary, keeping JPA entities out of the HTTP
  surface.
- **Spring Data JPA** — derived queries, JPQL with optional filters,
  `@Lock(PESSIMISTIC_WRITE)`, `@Version` optimistic locking.
- **Bean Validation (Jakarta)** — declarative request validation on DTO
  records, `@Valid` on controllers, centralised 400 mapping.
- **Global exception handling** — `@RestControllerAdvice` translating
  domain `ApiException`, `MethodArgumentNotValidException`,
  `AccessDeniedException` into stable JSON error envelopes.

## 2. Security
- **Spring Security HTTP Basic** with bcrypt password hashing.
- **Custom `UserDetailsService`** backed by the JPA repository.
- **Role-based authorisation** — `hasRole("ADMIN")` matcher, method
  security enabled, public-vs-protected route partitioning.
- **Auth helpers** — `AuthHelper.currentUser()` central read of the
  Spring Security principal into a domain `User`.

## 3. Concurrency / correctness
- **Pessimistic row-locking** to serialize hold attempts on the same
  seat across concurrent transactions.
- **Deterministic lock order** (sort IDs ascending) to prevent
  deadlocks under symmetric contention.
- **Optimistic versioning** as a defence-in-depth check on
  `ShowSeat`.
- **Scheduled background job** (`@EnableScheduling` + `@Scheduled`)
  releasing expired holds.
- **Async dispatch** (`@EnableAsync` + `@Async`) for notifications,
  keeping the booking critical path off I/O.

## 4. Domain modelling
- Mapping a real-world booking system into normalised entities — city,
  theater, screen, seat layout, movie, show, show-seat, booking,
  payment, discount, refund policy.
- Modelling **pricing tiers** as a combination of seat category +
  show-level tier multiplier — kept simple but expressive.
- Designing the **refund policy** as a tiered "if you cancel ≥ N hours
  before show, get P% back" so that admins can build cliffs without
  schema changes.

## 5. Persistence
- MySQL 8 / MariaDB schema generated by Hibernate, validated against
  a real MySQL on smoke-test.
- H2 in MySQL mode for fast in-memory test runs.
- Time stored in UTC (`hibernate.jdbc.time_zone`) with
  `LocalDateTime` boundaries on the API.
- Unique constraints and composite indexes on `show_seats` for both
  correctness (unique `(show, seat)`) and query speed
  (`status, hold_expires_at` for the expiry scheduler).

## 6. API design
- REST verbs and status codes used semantically (201 on create, 409 on
  conflict, 403 on RBAC denial, 400 on validation).
- Self-describing JSON DTOs using Java 17 records — no anaemic getters,
  no Lombok in the public surface.
- Predicate-style search query parameters (`/api/catalog/shows?cityId=&movieId=&date=`).

## 7. Testing
- **JUnit 5** + **Spring Boot's `@SpringBootTest`** for full-stack
  integration tests.
- **MockMvc + spring-security-test** to exercise the HTTP + auth
  layers without binding to a real port.
- **Concurrent test** using `ExecutorService` + `CountDownLatch` to
  reproduce the racing-customers scenario and assert "exactly one
  succeeds, rest get 409".
- **Time-bounded property override** via `@TestPropertySource` to
  shorten the hold expiry so the expiry test can run in seconds.
- Unit tests on the pure pricing function so the math is locked in.

## 8. Build & tooling
- Maven multi-profile-free build, with Lombok excluded from the
  shaded jar via the Spring Boot plugin.
- Reproducible dependency management through the
  `spring-boot-starter-parent` BOM.

## 9. AI-assisted engineering
- Driving an AI agent with explicit guard-rails ("don't refactor
  beyond the brief", "fail fast on missing config").
- Inspecting compiler / test output and feeding precise failure
  context back to the agent rather than restarting the conversation.
- Locking design decisions (entity model, locking strategy) in
  natural language before any code was generated, so the assistant's
  output stayed coherent across files.
- Writing the docs *after* the build was green, so the README
  reflects the shipped code rather than a fictional plan.
